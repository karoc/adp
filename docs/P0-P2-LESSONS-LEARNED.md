## 成功经验

### 1. 多 Agent 并行协作模式 ⭐⭐⭐⭐⭐

**实践**:
```
Wave 1: 3 agents 并行 (quickstart, prefix, errors)
Wave 2: 3 agents 并行 (json, docs, help)
Wave 3: 3 agents 并行 (bug1, bug2, bug3)
Wave 4: 2 agents 并行 (ci-fixer, security-auditor)
```

**优点**:
- **时间效率**: 6 小时工作在 ~2 小时内完成
- **任务隔离**: 每个 agent 独立工作，互不干扰
- **风险分散**: 一个 agent 失败不影响其他
- **资源利用**: 充分利用并行能力

**适用场景**:
- 任务互相独立
- 每个任务有明确边界
- 需要快速完成大量工作

**最佳实践**:
```markdown
1. 明确定义每个 agent 的职责和产出
2. 提供详细的上下文和验证方法
3. 设置清晰的完成标准
4. 使用一致的命名和文件结构
5. 预先创建任务清单，避免上下文丢失
```

---

### 2. 验证驱动的开发流程 ⭐⭐⭐⭐⭐

**流程**:
```
需求分析 → 审计现状 → 制定计划 → 执行验证 → 发现问题 → 修复验证 → 闭环
```

**关键环节**:
1. **方案审计**: 在执行前审计计划的可行性
   - 避免了 expect/jq 工具依赖问题
   - 调整了不切实际的时间预估

2. **自动化测试**: 每个验证都有可运行的脚本
   - 可重复执行
   - 易于回归测试
   - 便于 CI 集成

3. **量化指标**: 使用具体数字而非主观评价
   - 可用性评分 (1-5)
   - 测试通过率 (%)
   - 改进幅度 (+48%)

**经验教训**:
- ✅ **先审计再执行**: 避免了大量返工
- ✅ **量化优于定性**: 具体数字更有说服力
- ✅ **自动化优于手动**: 脚本可重复、可维护

### 3. 文档与代码同步演进 ⭐⭐⭐⭐

**实践**:
- 验证脚本 + 详细报告 + 综合总结
- 每个发现都记录在案
- 问题清单包含修复方案

**价值**:
- **可追溯性**: 知道每个问题从何而来
- **可维护性**: 新人可以快速理解
- **可传承性**: 经验不会随人员流动而丢失

**文档结构**:
```
scripts/                     # 可执行的验证脚本
docs/verification/           # 详细验证报告
  ├── *-verification.md      # 单项验证报告
  └── discovered-issues.md   # 问题汇总
VERIFICATION-SUMMARY.md      # 综合总结
PLAN-FOLLOWUP.md             # 后续计划
```

### 4. 问题分类与优先级管理 ⭐⭐⭐⭐

**分类体系**:
```
严重性:
  🔴 Critical: 阻塞性问题，必须立即修复
  🟡 Warning: 重要但非阻塞
  🟢 Info: 信息性，可选修复

优先级:
  P0: 立即执行
  P1: 1-2 周内
  P2: 1-2 月内
  P3: 长期规划
```

**优点**:
- 资源有效分配
- 避免陷入细节
- 明确轻重缓急

**本次应用**:
```
Bug #1: 中等/P1 → 30分钟快速修复
Bug #2: 低/P2 → 1小时，已修复
Bug #3: 低/P2 → 1小时，已修复
```

### 5. 脱敏审计作为标准流程 ⭐⭐⭐⭐⭐

**流程**:
```
代码变更 → 提交前自审 → 自动化扫描 → 手动审查 → 审计报告
```

**检查清单**:
- [ ] API keys / secrets / tokens
- [ ] 真实邮箱、电话、IP
- [ ] 内部路径、主机名
- [ ] 公司内部信息
- [ ] 未发布的产品细节

**工具组合**:
```bash
# 自动化扫描
git diff | grep -iE "(password|secret|key|token)"
git diff | grep -E "(\d{1,3}\.){3}\d{1,3}"
git diff | grep -E "[a-z0-9._%+-]+@[a-z0-9.-]+\.[a-z]{2,}"

# 手动审查
- 阅读所有 diff
- 检查提交信息
- 验证示例数据
```

**价值**:
- 防止敏感数据泄漏
- 符合安全合规要求
- 建立安全意识

## 遇到的挑战与解决方案

### 挑战 1: 任务上下文丢失

**问题**: 
对话很长后，TaskUpdate 失败，任务列表上下文丢失

**解决方案**:
- 不依赖持久化的任务列表
- Agent 完成后直接报告，由 team-lead 记录
- 使用 Git 提交作为进度追踪

**改进建议**:
- 定期保存任务状态到文件
- 使用更简单的进度追踪方式
- 减少对工具状态的依赖

---

### 挑战 2: CI 双语文档检查过严

**问题**:
技术报告也被要求有中文版本，导致 CI 失败

**解决方案**:
- 豁免内部技术文档目录
- 区分面向用户的文档和内部文档
- 更新 CI 检查规则

**经验**:
- 政策需要灵活性
- 不同文档类型应有不同要求
- 自动化检查需要豁免机制

---

### 挑战 3: 验证脚本初期的小问题

**问题**:
- prefix-verifier: 测试用例创建了歧义前缀
- json-verifier: 字段名错误（checks vs diagnostics）

**解决方案**:
- Agent 自行修复脚本
- 重新运行验证
- 更新报告

**经验**:
- 第一次运行失败是正常的
- 给 agent 足够的调试权限
- 迭代比完美更重要

### 常见陷阱

#### 陷阱 1: 过度设计验证方案
**症状**: 计划阶段耗时过长，方案复杂难执行
**解决**: 先做简单可行的方案，迭代改进

#### 陷阱 2: 忽视自动化验证
**症状**: 手动测试难以重复，回归问题频发
**解决**: 优先投入时间写验证脚本

#### 陷阱 3: 缺少量化指标
**症状**: 无法证明改进效果，决策缺乏依据
**解决**: 建立基线指标，对比前后变化

#### 陷阱 4: 文档与代码脱节
**症状**: 文档过时，实际行为与文档不符
**解决**: 自动化检查文档示例，CI 中验证

#### 陷阱 5: 忽略脱敏审计
**症状**: 敏感数据泄漏到公开仓库
**解决**: 提交前必须进行脱敏审计
